# Project Configuration (`.cosmon/config.toml`)

`.cosmon/config.toml` is the machine-readable source of truth for every
per-project setting cosmon needs. It is generated by `cs init` with every
section present (commented out where optional) so users discover the full
configuration surface without reading cosmon source code.

```
.cosmon/
├── config.toml          ← this file
├── surfaces.toml        ← surface projections
├── formulas/            ← canonical formulas (tracked)
├── molecules/           ← molecule declarations (tracked)
└── state/               ← runtime state (git-ignored)
```

## Schema overview

| Section | Purpose | Required |
|---------|---------|----------|
| `[project]` | Project identity (`project_id`) | **yes** |
| `[worker]` | Worker behavior on completion | no |
| `[surfaces]` | Auto-reconcile policy | no |
| `[documentation]` | Doc generation toggle | no |
| `[hooks]` | Lifecycle hook commands | no |
| `[gates]` | Verification gate commands (language-agnostic) | no |

All sections except `[project]` are optional and default to sensible
values. Missing sections never produce an error.

> **`[adapters].default` resolution chain (Q5a, `task-20260530-c089`,
> extended by `task-20260531-c99e`).** `cs tackle` picks its adapter,
> highest priority first: `--adapter` flag → formula-step `adapter` pin →
> `$COSMON_DEFAULT_ADAPTER` env var → this per-galaxy `[adapters].default`
> → global `~/.config/cosmon/config.toml` `[adapters].default` → built-in
> `"local"` floor. The env var is the operator's session-wide hammer (an
> empty value is treated as unset); the global file is the machine-wide
> preference, consulted only when this per-galaxy config declares no
> default. With nothing set anywhere the floor is always `local`, never a
> vendor cloud.

## `[gates]` — project verification gates

`[gates]` defines the shell commands a worker should run to prove its
change is green. Every field is optional and cosmon treats each command
as an opaque string — **no toolchain is assumed**. A Python, Node, Go,
or Rust project configures the same schema, just with different command
strings.

```toml
[gates]
build_command     = "cargo check --workspace"
test_command      = "cargo test --workspace"
lint_command      = "cargo clippy --workspace -- -D warnings"
format_command    = "cargo fmt --all -- --check"
typecheck_command = "mypy ."           # optional standalone type-check
setup_command     = "uv sync"          # dependency install / bootstrap
doc_command       = "RUSTDOCFLAGS='-D warnings' cargo doc --workspace --no-deps"
integrity_command = "cargo check --workspace --all-targets"  # cs done merge gate
fail_closed_on_unverified = false      # also roll back the "nothing declared" case (expected:true already fails closed)
```

### Fields

| Field | Purpose |
|-------|---------|
| `setup_command` | Prepare the worktree (install dependencies, sync virtualenv) |
| `build_command` | Compile / type-check the project (fast) |
| `test_command` | Run the test suite |
| `lint_command` | Run the linter with errors-as-failures |
| `format_command` | Verify formatting without rewriting files |
| `typecheck_command` | Standalone type-checker (when the build step does not subsume it) |
| `doc_command` | Build the API docs with warnings promoted to errors. Not subsumed by the others: `build_command` compiles code without resolving a doc link, and a code linter is not a doc linter, so a broken intra-doc link passes every other gate and surfaces only in CI on the trunk. |
| `integrity_command` | The `cs done` **post-merge integrity gate** — run on the combined tree after a branch merges (ADR-158). See below. |
| `fail_closed_on_unverified` | Bool (default `false`). Promote an `Unverified` post-merge outcome from loud-fail-open to a rollback. See below. |

### `integrity_command` and the post-merge cascade (`cs done`, ADR-158)

`cs done` runs a fast integrity gate on the **combined** tree after a branch
merges but before the landing is observable to the scheduler — if it fails, main
is reset to its pre-merge revision and the DONE is refused. Cosmon owns *when*
to run it; the galaxy declares *what* it means, through a cascade (highest wins):

1. `integrity_command` — run verbatim from the repo root; exit 0 = verified,
   non-zero = the merge is rolled back. The *fast* combined-compile/type-check
   tier (hard-bounded, **not** a test run). Examples:
   `cargo check --workspace --all-targets`, `python -m compileall . && mypy .`,
   `go build ./... && go vet ./...`.
2. cargo-metadata auto-detect — the zero-config path when `integrity_command` is
   unset and the repo is a Cargo workspace cargo resolves from the root.
3. `build_command` — the polyglot fallback when neither above applies.
4. otherwise a loud, durable `Unverified` witness — cosmon has (or cargo has) no
   way to verify this tree. What happens next follows the **D1 discriminator**
   (`fail_closed = expected || fail_closed_on_unverified`):
   - **A gate was expected** (cargo resolved the workspace, or a command was
     declared) but a code diff still went unchecked → **fail closed by default**:
     main is reset, the DONE is refused, `error:` witness, non-zero exit. This
     protects cosmon's own net with no opt-in.
   - **Nothing was declared** to verify the tree → the merge lands with a durable
     `ok:unverified` witness in `events.jsonl` (out loud, never a clean-`ok` lie),
     *unless* `fail_closed_on_unverified = true` promotes it to a rollback.

Leave `integrity_command` unset to ride the cargo auto-detect (cosmon's own
config does this). `integrity_command` / `build_command` are repo-supplied shell,
so `cs done` refuses to exec them in an untrusted clone (`cs trust` is the gate)
**and runs them under the egress jail** — the same sandbox as an ordinary agent
subprocess. On a host that cannot kernel-enforce a required `deny-external`
policy for an exposed multi-tenant dispatch, the gate refuses fail-closed rather
than run the shell unconfined (see ADR-158 §4).

`fail_closed_on_unverified = true` promotes the **nothing-declared**
(`expected:false`) `Unverified` outcome to a rollback (`error:` witness, non-zero
exit) — for an operator who wants "no declared verification ⇒ no merge". The
expected-but-unchecked (`expected:true`) case already fails closed by default, so
this flag is about the residual "nobody declared a verifier" case.

`cs validate` (the heavyweight tier-2 milestone gate) delegates symmetrically to
`test_command` / `lint_command` / `format_command` / `typecheck_command` /
`doc_command`, falling back to the cargo defaults when a slot is unset, and runs its
repo-supplied stages under the same egress jail. The `doc_command` fallback is
the one exception to "unset ⇒ cargo default": it applies only when the repo root
actually holds a `Cargo.toml`, so a polyglot galaxy silent on the slot gets no
doc stage rather than a `cargo doc` it cannot run.

### Language examples

| Stack | `build_command` | `test_command` | `lint_command` | `format_command` |
|-------|-----------------|----------------|----------------|------------------|
| Rust | `cargo check --workspace` | `cargo test --workspace` | `cargo clippy --workspace -- -D warnings` | `cargo fmt --all -- --check` |
| Python | `uv sync` | `pytest` | `ruff check .` | `ruff format --check .` |
| Node | `npm run build` | `npm test` | `eslint .` | `prettier --check .` |
| Go | `go build ./...` | `go test ./...` | `golangci-lint run` | `gofmt -l .` |

### Integration with `cs tackle`

When `cs tackle` dispatches a worker, it loads `.cosmon/config.toml` and
renders the configured gates into the bootstrap prompt.

- **All fields unset** → the prompt falls back to a neutral instruction:
  *"Run the project's verification gates (see `.cosmon/config.toml`
  `[gates]` or the project's CLAUDE.md)."*  This is language-agnostic and
  never breaks a non-Rust project.
- **Some fields set** → the prompt renders exactly those commands as a
  numbered list, in the canonical order
  `setup → build → typecheck → test → lint → format`. The worker runs
  exactly what the project author configured — no more, no less.

This replaces a previous Rust-biased hardcoded line in the worker prompt
(`cargo check && cargo test && cargo clippy -- -D warnings && cargo fmt
--check`), which silently broke Python, Node, and Go projects.

## `[hooks]` — lifecycle hook commands

```toml
[hooks]
pre_done   = "tools/ci/verify-functional-evidence.sh"
post_merge = "just install"
```

| Field | When it runs | On non-zero exit |
|-------|--------------|------------------|
| `pre_done` | **Before** `cs done` merges a worker branch (blocking gate) | **Aborts** teardown — nothing merged or torn down |
| `post_merge` | **After** `cs done` successfully merges a worker branch | Warns only — the merge already landed |

Both hooks run from the repository root. The asymmetry in the last column
is structural, not stylistic: a hook may abort only while the operation it
guards is still reversible. `pre_done` runs before the merge (reversible),
so it can *refuse* a DONE; `post_merge` runs after (irreversible), so it can
only warn.

### `pre_done` — the blocking Definition-of-Done gate

`pre_done` lets a galaxy make a **falsifiable Definition-of-Done enforceable
from inside the molecule cycle** — "a rigorous workflow must be able to
refuse a DONE it cannot prove". Before this hook, `cs done` had no way to
reject a merge; a DoD could only be enforced out-of-band in GitHub
branch-protection, outside cosmon's teardown.

- The command is invoked as `sh -c '<pre_done>' -- <molecule-id>`, so the
  molecule under teardown is reachable as `$1` and the script can scope its
  verification to that molecule.
- **Exit 0** → teardown proceeds (merge, worktree removal, branch delete,
  tmux kill).
- **Exit non-zero** → `cs done` returns a hard error carrying the script's
  stderr, and the merge, worktree, branch, and tmux session are all left
  untouched. Fix the gap and rerun.
- **Idempotent**: the hook is expected to be a pure read of repository /
  molecule state, so a failed-then-retried `cs done` re-runs it identically.
- **Kill-switch** (human operator only): `cs done --skip-pre-done-hook`, or
  the `COSMON_SKIP_PRE_DONE_HOOK` environment variable set to any non-empty
  value. Reserved for a deliverable the operator knows is good but the script
  cannot see (e.g. evidence living outside the repo).
- Ships **absent by default** — every existing project is unaffected.

## `config.toml` vs `CLAUDE.md` — overlap and source of truth

Cosmon's own `CLAUDE.md` has historically documented Gas Town Rig
Commands (`build_command`, `test_command`, …) as markdown prose. With
`[gates]` these values now also live in `.cosmon/config.toml`, and the
two can drift.

The policy is:

- **`.cosmon/config.toml` `[gates]` is the machine source of truth.**
  Tools — `cs tackle`, Gas Town bridges, external orchestrators — read
  the TOML.
- **`CLAUDE.md` documents for humans** and should cross-reference
  `.cosmon/config.toml` rather than repeat the commands. If you do
  repeat them for narrative reasons, treat the TOML as authoritative
  and keep the two in sync when you change one.

This is the same rule git itself applies to its `config` file vs the
human-readable docs in `Documentation/`: the file is the source of
truth; the prose is navigation.
