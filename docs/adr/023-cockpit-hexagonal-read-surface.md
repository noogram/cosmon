# ADR-023: Cockpit — Hexagonal Read Surface with Subprocess Write Path

## Status
Accepted (2026-04-10)

## Context

Cosmon needs a live dashboard for fleet observation — molecule state, worker
liveness, zombie detection. The question is: what kind of thing is this
dashboard in the architecture?

Two framings were considered:

1. **Projection** (like `STATUS.md`, `ISSUES.md`). The dashboard is a surface
   generated by `cs reconcile`, a snapshot written to disk and served
   statically. This is the pattern of ADR-017 and the surface sync protocol.

2. **Live read view** (retina). The dashboard reads the state store directly
   on every request and computes its own view-model. It never writes state.
   It is not a projection — it is an observation instrument.

The projection model is wrong for the dashboard. Projections are reconciled
at discrete points in time and can drift between reconcile runs. A dashboard
that shows stale data is worse than useless — it erodes trust in the system.
The dashboard must read live state, not a snapshot.

Wheeler framing: **the dashboard is Cosmon's retina, not its window.** A retina
receives light from the world and translates it into a form the observer can
process. It does not emit light. It does not alter what it observes. The
cockpit observes the state store; it does not participate in domain logic.

The remaining question: what about writes? The dashboard has a "Spark" input
box — users type an idea and the cockpit nucleates a molecule. This write
path must not bypass the transactional core (ADR-016). The cockpit must not
directly mutate `.cosmon/state/`. Instead it delegates to `cs nucleate` via
subprocess — the same path a human at a terminal would use.

## Decision

### 1. The cockpit is a live read view, not a projection

The cockpit reads `.cosmon/state/` directly on every HTTP request via
`FileStore` + `StateStore`. It computes view-model types (`MoleculeSummary`,
`MoleculeDetail`, `FleetSummary`, `Revision`) and serves them as JSON.
There is no reconcile step, no snapshot, no surface file.

The cockpit is **stateless between requests**. It holds no in-memory cache,
no WebSocket state, no session. Every request is a fresh read from disk.
This makes the cockpit trivially restartable and immune to state corruption.

### 2. The only write path is subprocess `cs nucleate`

The `POST /api/spark` endpoint spawns `cs nucleate` as a child process.
This preserves the ADR-016 transactional-core invariant: all state mutations
go through the `cs` CLI, which reads-mutates-writes in a single invocation.

```
POST /api/spark { "text": "..." }
  → tokio::process::Command::new("cs")
      .args(["nucleate", "idea-to-plan", "--kind", "idea",
             "--var", &format!("topic={text}"), "--json"])
  → 200 OK { molecule JSON from cs stdout }
```

The cockpit never imports state-mutation functions. It never calls
`StateStore::save_molecule`. The subprocess boundary is the firewall.

### 3. Two crates, port/adapter separation

```
cosmon-cockpit          (port crate — zero I/O, zero cosmon-core in the port)
├── view.rs             DashboardView trait, DTOs, Liveness, CockpitError
│                       All types use plain String for status/kind — no core enums
├── adapter.rs          FileCockpitView: maps cosmon-core types → string DTOs
│                       (this module imports cosmon-core for the mapping)
└── lib.rs              Public API

cosmon-cockpit-http     (adapter crate — axum binary)
├── main.rs             HTTP handlers, worker liveness probing, spark subprocess
└── static/index.html   Embedded HTML dashboard (auto-refresh every 2s)
```

**The `DashboardView` trait and all DTO types are free of `cosmon-core`
imports.** Status and kind are plain `String` values, not domain enums.
This keeps the port boundary clean: any future adapter (runtime cockpit,
test mock, alternative backend) can implement `DashboardView` without
importing the core domain model.

The `FileCockpitView` adapter *does* import `cosmon-core` to map domain
types to strings. This is appropriate — the adapter is the integration
seam. The port contract remains core-free.

### 4. Cross-sourced liveness (zombie detection)

The cockpit enriches molecule data with worker liveness from two sources:

- **Transport state**: probes tmux sessions via `TmuxBackend::is_alive()`.
- **Cognitive state**: reads worker self-declaration from
  `.cosmon/state/cognitive/{worker_id}.json`.

The two signals are composed into a `Liveness` verdict:

| Molecule status | Worker signal | Verdict |
|-----------------|---------------|---------|
| `running` | `working*`, `loading*` | `healthy` |
| `running` | `waiting`, `done` | `mismatch` |
| `running` | `idle`, `dead`, `stale`, `error*`, `-`, empty | `zombie` |
| `running` | (no worker) | `unknown` |
| non-running | any | `unknown` |

`compute_liveness()` takes plain string arguments (`&str` for status,
`Option<&str>` for worker live), keeping it independent of core enums.

### 5. Future: `RuntimeCockpitView`

When the resident runtime ships (ADR-016 Phase 3+), a second adapter will
implement `DashboardView` by querying the runtime's in-process state rather
than reading files from disk. The trait contract is designed to accommodate
this:

- `DashboardView` is object-safe (no generics, no `Self: Sized` bounds).
- The HTTP server holds `Arc<dyn DashboardView>` — the concrete adapter
  is selected at startup based on whether the runtime is available.
- The runtime adapter reads from the same state store but can add
  runtime-specific signals (DAG progress, policy decisions, event log).

This is the "retina upgrade" — same optic nerve (trait), sharper sensor
(runtime adapter).

## Consequences

**Positive:**

- Dashboard always shows live state. No reconcile lag, no stale snapshots.
- Write path goes through the transactional core. No backdoor mutations.
- Port types are decoupled from domain enums. Alternative adapters
  (test mocks, runtime view) are trivial to implement.
- Stateless between requests — crash-safe, trivially restartable.
- Zombie detection catches stuck workers in ≤2 seconds (polling interval).

**Negative:**

- Every HTTP request reads from disk. For large fleets (100+ molecules)
  this may become a bottleneck. Mitigation: the runtime adapter (§5) can
  serve from memory when available.
- The subprocess write path adds ~50ms latency to spark nucleation.
  Acceptable for a human-facing UI; unacceptable for high-throughput
  automated nucleation (which should use `cs nucleate` directly).
- `cosmon-cockpit` crate still has `cosmon-core` in its dependency tree
  (via the `FileCockpitView` adapter). The port boundary is clean but the
  crate-level dependency exists. A future extraction of `view.rs` into
  `cosmon-cockpit-api` would achieve full crate-level decoupling.

**Neutral:**

- The HTML dashboard is embedded in the binary via `include_str!`. This
  is simple but means UI changes require recompilation. A future
  `--dev` flag could serve from the filesystem instead.
