# Contributing to Cosmon

Welcome. Cosmon is a stateless Rust CLI that gives AI coding agents a
persistent identity, a typed lifecycle, and crash-recovery. It is early-stage
but functional — the core types, the `cs` CLI, state persistence, tmux
transport, and the DAG engine all ship today. The API is still moving, so
expect breaking changes before 1.0.

## Start here

The full contributor guide — quality rules, testing tiers, vocabulary, the
Rust rules CI enforces — lives in **[docs/CONTRIBUTING.md](docs/CONTRIBUTING.md)**.
Read it before opening a PR. This page is the short front door.

Two design documents govern every change:

- **[THESIS.md](THESIS.md)** — *how* the agentic runtime works (typestate,
  newtype, zero-I/O core, physics vocabulary).
- **[docs/architectural-invariants.md](docs/architectural-invariants.md)** —
  the operational discipline that preserves the system's structural
  mechanisms. Run the coherence checklist there before adding or renaming a
  command. *If the code contradicts the thesis or the invariants, the thesis
  wins — file an issue, do not backdoor the change.*

For the operator mental model (lifecycle, DAG, tags, monitoring), read
[docs/handbook.md](docs/handbook.md) or run `cs help guide`.

## Definition of Done

Every PR must pass all four gates locally before review:

```bash
cargo check --workspace                    # build
cargo test --workspace                     # test
cargo clippy --workspace -- -D warnings    # lint
cargo fmt --all -- --check                 # format
```

The clippy gate is non-negotiable regardless of which testing tier applies to
the code you touched (see the testing policy in
[docs/CONTRIBUTING.md](docs/CONTRIBUTING.md)).

> **Do not run `just install` from a feature branch / worktree.** Installation
> happens only on the trunk after a change lands. Verify your work with the
> four `cargo` gates above — do not deploy. `just install` builds and installs
> only the public cosmon product binaries; the private federation tooling
> (`neurion`, `topon-mcp`, `almanac`) is an operator-workstation concern behind
> `just install-federation` and is never part of the contribution loop.

## Finding work

Cosmon tracks its own work as **molecules**, not GitHub issues. The current
backlog is projected to two human-readable surfaces at the repo root,
regenerated by `cs reconcile`:

- **[ISSUES.md](ISSUES.md)** — every open 🐛 issue and pending molecule.
- **[STATUS.md](STATUS.md)** — fleet-wide state at a glance.

Pick something from `ISSUES.md`, or open a regular GitHub issue / discussion to
propose new work.

## Naming: Noogram vs cosmon

Two names, one rule: **Noogram = what you read about / install (the
distribution); cosmon = what you run (the kernel, the `cs` binary).** When a
sentence is borderline, apply the swap test in
[docs/style/naming.md](docs/style/naming.md) — replace the word with "the `cs`
binary"; if it still reads, it was *cosmon*, otherwise *Noogram*. The same doc
records the **one gate**: a tool earns a public doc section or a
`/<tool>/install.sh` endpoint only when a stranger can install it. That gate is
enforced mechanically by `scripts/check-docs-one-gate.sh` (a CI job), so the
confidential neurion product can never surface and no premature tool can grow an
install page.

## Git rules

- **Conventional Commits** (strict): `feat:`, `fix:`, `refactor:`, `test:`,
  `docs:`, `chore:`.
- **PR max 400 lines** — split larger changes into stacked PRs.
- **CHANGELOG.md** updated with every user-visible change (Keep a Changelog
  format).
- **`Co-Authored-By`** trailer for AI-assisted commits.
- **Sign off every commit** with `git commit -s` — the DCO (see below).

## License

Cosmon is a mixed-licence workspace (see
[ADR-092](docs/adr/092-license-bascule-mpl-to-agpl.md)): the core — kernel,
persistence, runtime, agent engine, cockpit, and the multi-tenant server,
including `cosmon-core` — is AGPL-3.0-only ([LICENSE](LICENSE)). Only the pure
network SDK a third party links in *to talk to* a cosmon instance over the wire
(`cosmon-client`, `cosmon-thin-cli`, `cosmon-thin-macro`) is
Apache-2.0 ([LICENSE-APACHE](LICENSE-APACHE)); each crate's `license` field in
its `Cargo.toml` is the ground truth. By contributing you agree your
contribution is licensed under the same terms as the file you touch.

## Sign your work — Developer Certificate of Origin (DCO)

Cosmon uses the [Developer Certificate of Origin](DCO.md) (DCO 1.1) — the
Linux-kernel model: a lightweight, per-commit certification that you have the
right to submit your contribution under the project's licence. We use the DCO
**rather than a CLA on purpose**: it certifies *provenance* without
concentrating rights in a single holder, which is what keeps the commons
non-capturable. Nothing is assigned — your contribution stays under the same
terms as the file you touch.

**Stewardship.** Cosmon is the open kernel of the Noogram commons. Governance is
being transferred to the **Noogram association** (in formation); once effective,
the association — not any individual — is the steward that holds licence
enforcement and the "main stays public" ratchet. A clean DCO record is what
makes that transfer sound.

**How to sign.** Add a `Signed-off-by` trailer to every commit:

```bash
git commit -s -m "feat: your change"
```

This appends `Signed-off-by: Your Name <your@email>` from your `git config`
identity, certifying the [DCO](DCO.md). A DCO check gates every PR — unsigned
commits block the merge. Fix a missed sign-off with `git commit --amend -s`, or
`git rebase --signoff <base>` for a range.

**Never want to type `-s`?** Activate the repo's `prepare-commit-msg` hook — it
appends the trailer automatically on every commit (idempotent):

```bash
ln -sf ../../hooks/prepare-commit-msg .git/hooks/prepare-commit-msg
```

The symlink form composes with any other local hooks; prefer it over
`core.hooksPath`.

## Release integrity

Release tags are **cryptographically signed** with a hardware key (SSH signing
via a FIDO2 security key, touch-to-sign) and show as **Verified** on GitHub.
Verify any release locally:

```bash
git tag -v v0.1.0    # expects: Good "git" signature for <maintainer>
```

This is distinct from the DCO sign-off: the sign-off certifies the *right* to
contribute; the tag signature proves the release genuinely comes from the
maintainer, on a key that never leaves the hardware. Contributors are **not**
required to cryptographically sign their commits — only the DCO sign-off above.
A project about verifiable provenance signs its own artifacts.
