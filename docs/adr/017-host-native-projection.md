# ADR-017: Host-Native Projection and Surface Rendering Invariants

## Status
Proposed (2026-04-09)

## Context

Cosmon projects molecule state onto surfaces the host project already
owns: `STATUS.md`, `ISSUES.md`, `docs/adr/INDEX.md`, and GitHub Issues.
The original projection code treated cosmon as a co-author of those
surfaces: every rendered file carried a `<!-- Generated by cosmon -->`
header, every GitHub Issue body opened with a **Molecule / Formula /
Fleet** metadata block, and every footer ended with *"Projected by
cosmon surface"*. On a shared repository, that turned routine tickets
into cosmon advertisements.

A panel (feynman, jobs, wheeler) convened as
`delib-20260409-f4e1` to dissect the rendering choices. Synthesis #7
— the one this ADR encodes — crystallised three independent
decisions that had been tangled together in the code:

1. **Backref marker stability.** The HTML marker
   `<!-- cosmon:molecule:ID -->` is the idempotency key the projection
   engine uses to find an existing GitHub Issue and update it in
   place. If the writer ever changed the marker format, every
   pre-existing issue would lose its anchor and be duplicated on the
   next reconcile. The marker must therefore be **write-once,
   read-forever**.
2. **Host-native vs attributed rendering.** There are two legitimate
   audiences for a projected surface, and the code was trying to
   serve both with a single template. Shared-repo surfaces (the
   GitHub tracker of an unrelated open-source project, a STATUS.md
   committed to somebody else's repo) want the host to own the
   artefact; cosmon is the tool that produced it, not a co-owner.
   Cosmon-owned surfaces (anything under `.cosmon/`, the future
   `docs/adr/`-internal dashboards, the operator's own repos) can
   announce themselves because the reader already knows what cosmon
   is.
3. **Mirror schema evolution.** The local GitHub mirror
   (`.cosmon/state/surfaces/github/{repo}/{mol}.json`) records what
   cosmon last projected so the sync is skip-if-unchanged. As the
   body renderer and backref policy evolve, the mirror schema must
   be versioned — and old mirrors must upgrade gracefully rather
   than trigger a forced reproject of every issue.

Wheeler's reframe on transparency (see THESIS Part XVI amendment)
sharpened the first two points: **transparency is a property of the
surface declaring it is auto-generated and pointing at its source, not
of the surface crediting the rendering engine.** A `Makefile`-built
`.o` file is transparent without being stamped *"built by GCC"*.
Cosmon surfaces should follow the same discipline.

## Decision

### 1. The backref invariant (additive marker policy)

The HTML comment marker embedded in every projected GitHub Issue body
is a permanent foreign-key anchor. Three invariants govern it:

- **The writer never rewrites an existing marker.** When the
  projection engine updates an issue, it must preserve the exact
  marker string it originally wrote, even if cosmon's current
  default marker format has changed.
- **The reader accepts `cosmon:molecule:ID` forever.** The search
  pattern (`gh issue list --search "cosmon:molecule:ID in:body"`)
  must remain valid as a stable recognition key across all cosmon
  versions. No future rename (`cosmon:m:`, `cx:molecule:`, etc.)
  may remove support for the original form.
- **New markers may be added; old markers may not be removed.**
  If a future version adds a richer marker (e.g. an additional
  `cosmon:fleet:FID` anchor), it is appended alongside the original,
  not in place of it. The body may carry multiple cosmon markers;
  the reader takes the union.

This is an **additive marker policy**: the set of recognised markers
only grows over time. The rationale is operational — cosmon is used
against long-lived third-party issue trackers where a single bad
reconcile can spawn hundreds of duplicates, and recovery is manual
and tedious.

The marker is classified as **invisible plumbing**: GitHub never
renders HTML comments in issue bodies, so it imposes zero visible
cosmon vocabulary regardless of branding mode. It is the one
exception to the "no cosmon strings in host-native mode" rule.

### 2. Host-native is the default for shared surfaces

Surface rendering has two modes, declared per-surface in
`surfaces.toml` via the `branding` field:

| Mode           | Metadata block | Footer                                                          | Header                                                          | Default for                        |
|----------------|----------------|-----------------------------------------------------------------|-----------------------------------------------------------------|------------------------------------|
| `host-native`  | dropped        | `<!-- auto-generated from {dir_name}/ — edit the source -->`    | `<!-- auto-generated from {dir_name}/ — edit the source -->`    | **All shared-surface projections** |
| `attributed`   | full           | `*Projected by cosmon surface. Source of truth: \`.cosmon/\`*`   | `<!-- Generated by cosmon. Source of truth: .cosmon/ -->`      | cosmon-owned surfaces (`.cosmon/-internal`) |
| `none`         | dropped        | (no footer)                                                     | (no header)                                                     | Embedded snippets inside larger docs |

**Default policy.** When a surface does not specify `branding`, it
defaults to `host-native`. Rationale: the vast majority of
projections land on repositories the operator shares with people who
have never heard of cosmon. The tool should be invisible in the
default case and visible only on explicit opt-in.

**Attributed is reserved for cosmon-owned surfaces.** The canonical
use case is dashboards that live under `.cosmon/-internal/` (future
resident-runtime views, operator-only cockpit files). A surface
under `.cosmon/` *may* still be rendered host-native — the rule is
directional: only cosmon-owned surfaces are allowed to choose
attributed, never the reverse.

**`{dir_name}` in the footer is the source directory the surface
projects from**, not literally `.cosmon/`. For example, a rendered
`docs/adr/INDEX.md` footer says *"auto-generated from docs/adr/ —
edit the source"*, pointing at the directory whose contents drive
the projection, not at cosmon's internal state directory. The reader
is told where to make the change, not what tool produced the file.

**Minimum footer.** The host-native footer is a single HTML comment:

```
<!-- auto-generated from {dir_name}/ — edit the source -->
```

It must contain (a) the words "auto-generated" so tools detecting
generated files can match on it, (b) the source directory so a human
can navigate to where to make a change, and (c) no cosmon
vocabulary. Anything more is a violation of host-native mode.

### 3. `MIRROR_SCHEMA_VERSION = 2` with graceful v1 upgrade

The local GitHub Issues mirror is bumped to schema version 2. The
version is stamped in each `{molecule-id}.json` file under a new
top-level `schema_version: u32` field.

**v1 mirrors** (no `schema_version` field, or `schema_version == 1`)
record: `molecule_id`, `issue_number`, `repo`, `title`, `body_hash`,
`state`, `kind`, `status`, `projected_at`. They were written before
the host-native default existed, so their `body_hash` reflects the
attributed render of the body.

**v2 mirrors** add: `schema_version = 2`, plus `branding` (the mode
used when the body was rendered) and `marker` (the exact marker
string embedded, so the writer can obey the never-rewrite invariant
even if cosmon's current default marker format drifts in a future
version).

**Graceful upgrade rules.**

1. **Reading a v1 mirror never fails.** Missing fields deserialise to
   safe defaults (`schema_version = 1`, `branding = attributed`,
   `marker = format!("<!-- cosmon:molecule:{} -->", molecule_id)`).
2. **A v1 mirror does not force a reproject.** The skip-if-unchanged
   path compares `body_hash`; if the current render matches the v1
   hash under the v1 branding, the issue is skipped.
3. **The first write after upgrade promotes the mirror to v2.** When
   the projection engine writes any mirror — whether the issue body
   changed or not, as long as a write is happening for some other
   reason — it writes it in v2 format.
4. **Branding mode changes count as a body change.** Flipping a
   surface from `attributed` to `host-native` will invalidate the
   v1 hash on the next reconcile and trigger a one-time reproject.
   The reproject honours the backref invariant: the existing marker
   from the v1 body (reconstructed via the default format) is
   preserved in the new v2 body.
5. **Unknown future schema versions are rejected loudly.** A mirror
   with `schema_version > 2` causes the reconcile to abort with an
   error pointing the operator at the cosmon upgrade docs, rather
   than silently downgrading.

### 4. Non-decisions (explicitly out of scope)

- **Template pluggability** — the three rendering paths
  (`attributed`, `host-native`, `none`) are hard-coded. Users cannot
  supply arbitrary templates. This is revisited in ADR-018.
- **Per-molecule branding override** — branding is a property of
  the surface, not the molecule. A molecule rendered onto a
  host-native surface is host-native regardless of its kind or
  metadata. This keeps the mental model regime-consistent.
- **GitHub marker evolution** — the additive policy forbids removal,
  but this ADR does not schedule any new marker. None is needed
  today.

## Consequences

**Positive:**

- Shared-repo projection no longer leaks cosmon jargon into issue
  trackers operators don't own. The tool disappears into the
  artefacts it produces.
- The backref invariant is now an explicit contract instead of an
  implicit property of the current implementation. Future refactors
  have a fixed point they must respect.
- Schema versioning on the mirror makes body-format evolution safe.
  The mirror now stores *what* was rendered and *how*, not just the
  resulting hash.
- The `{dir_name}` pointer in the footer is actionable: a reader who
  wants to change the surface now knows where to look without first
  learning what cosmon is.

**Negative:**

- Two rendering modes is one more axis the rendering code has to
  respect. Every future feature (new sections, new footers, new
  metadata) must be tested in both modes.
- Operators who *want* cosmon to be visible on a shared repo now
  have to opt in explicitly. This is a deliberate friction, but it
  is a friction.
- The v1→v2 upgrade path adds a deserialisation branch that will
  have to be maintained for as long as any v1 mirror could plausibly
  exist in the wild. Eventually (ADR TBD) v1 support will be
  removed with a major-version bump.

**Neutral:**

- This ADR does not introduce new commands or new CLI flags. It is
  a policy layer over existing machinery — the
  `Branding::HostNative` default, the marker preservation logic,
  and the mirror schema field are implementation details. The ADR
  exists to pin the *contract* so it doesn't drift silently.
